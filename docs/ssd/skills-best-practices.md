# Skills活用の役割設計ベストプラクティス

このガイドは、Application実装で Skills を継続活用するための **役割分担・運用ルール・品質基準** を定義します。

## 1. 役割（最小5ロール）

### 1-1. Skill Owner（責任者）
- Skillの目的・適用範囲・優先順位を定義する。
- 変更の最終承認とリリース判断を行う。

### 1-2. Domain Expert（業務知識担当）
- ドメイン固有ルール、制約、用語集を提供する。
- Skill内の reference 情報の正確性を担保する。

### 1-3. Skill Engineer（実装担当）
- `SKILL.md`、`scripts/`、`references/`、`assets/` を実装・更新する。
- 再利用性を高めるため、処理をテンプレート化する。

### 1-4. QA Reviewer（品質担当）
- トリガー妥当性、手順の再現性、失敗時ハンドリングを検証する。
- サンプル入力に対して期待結果が得られるか確認する。

### 1-5. Ops Maintainer（運用担当）
- バージョン管理、リリースノート、廃止管理を実施する。
- 利用ログをもとに改善提案を行う。

## 2. RACI（推奨）

| 主要タスク | Owner | Domain Expert | Skill Engineer | QA Reviewer | Ops Maintainer |
|---|---|---|---|---|---|
| Skill企画・優先度決定 | A/R | C | C | C | C |
| 要件/制約整理 | A | R | C | C | I |
| Skill実装 | C | C | A/R | C | I |
| 検証・受け入れ | A | C | C | R | I |
| リリース・監視 | A | I | C | C | R |

- A: Accountable（最終責任）
- R: Responsible（実行責任）
- C: Consulted（相談先）
- I: Informed（共有先）

## 3. Skill設計のベストプラクティス

1. **トリガー条件を明確化する**
   - `description` に「何をするSkillか」だけでなく「いつ使うか」を含める。
2. **SKILL.mdは短く、詳細は references に分離する**
   - 運用手順は本体、詳細仕様や辞書は `references/` に置く。
3. **再実行可能な処理は scripts 化する**
   - 人手で繰り返す処理を減らし、結果の揺れを抑える。
4. **失敗時のフォールバックを先に書く**
   - 依存ファイル不足、外部接続不可などの代替手順を明記する。
5. **最小コンテキストで動くように設計する**
   - 大量文章の直書きより、必要時ロード前提の構成を優先する。

## 4. 運用フロー（推奨）

1. 目的と対象ユースケースを定義する（Owner主導）。
2. 業務知識を整理し、再利用資産を分類する（Domain Expert + Engineer）。
3. Skill実装（`SKILL.md` + 必要に応じて `scripts/`, `references/`, `assets/`）。
4. QAチェック（トリガー、手順再現性、エッジケース）。
5. リリース後に利用ログを見て改善チケット化する。

## 5. 受け入れ基準（Definition of Done）
- [ ] 役割（Owner/Expert/Engineer/QA/Ops）が割り当て済み。
- [ ] Skillのトリガー条件が `description` に明記されている。
- [ ] 代表ユースケースで再現可能な検証ログがある。
- [ ] 失敗時フォールバックが文書化されている。
- [ ] 改善サイクル（定期レビュー日）が決まっている。
